## 📌 프로젝트 소개

WMS(Warehouse Management System)는 창고 내 상품의 **입고, 보관, 출고 과정을 체계적으로 관리**하고, 이를 통해 **재고 현황과 물류 흐름을 효율적으로 통제**하는 시스템입니다.

본 프로젝트는 WMS의 핵심 기능들을 직접 구현하며, 시스템의 구조와 동작 원리에 대한 이해를 목표로 진행되었습니다.

주요 구현 기능은 다음과 같습니다:

- 제품의 입고 및 출고 처리
- 재고 수량 실시간 반영 및 변경 이력 기록
- 보관 조건(온도 등)에 따라 자동으로 보관 구역 배정
- 구역별 온도 설정 및 제어 기능 제공

이를 통해 **입출고 흐름의 체계화**, **자동화된 보관 관리**, **기능별 역할 분리** 등을 구현하며,

WMS의 **구성 요소 간 유기적인 연결**과 그에 따른 **전체 시스템의 통합적 작동 원리**를 직접 설계하고 이해할 수 있었습니다.

향후 2차 프로젝트에서는 기능 확장과 구조 고도화를 통해 시스템의 완성도를 더욱 높일 계획입니다

## ⚙ 프로젝트 내 역할 및 기여

### ✅ 보관 조건 기반 자동화 로직 설계

- **입고 요청이 들어오면, 해당 식재료의 보관 조건과 공간 크기를 고려해서 자동으로 구역을 배정하도록 만듦**
→ 사람이 직접 고르지 않아도, 시스템이 자동으로 적절한 위치를 찾아 배치하게 함

<br>

### ✅ 재고 및 재고 이력 관리 자동화

- **입고/출고가 일어날 때마다 자동으로 재고 수량이 바뀌고, 그 기록이 남도록 트리거를 사용해서 구현함**
    
    → 재고 수량이 실시간으로 반영되고, 변경 이력도 따로 관리되도록 처리함
    
- **입출고 내역을 콘솔에서 실시간으로 조회할 수 있게 만듦**
    
    → 어떤 요청이 언제, 어떤 상태로 처리됐는지 (승인/대기/취소) 바로 확인할 수 있도록 함
    

<br>

### ✅ Git 브랜치 전략 적용 및 관리

- **Git Flow 전략을 기반으로 `main`, `develop`, `feature`, `hotfix` 브랜치를 나누고, 협업 시 충돌 없이 개발할 수 있도록 전략을 세우고 운영함**
- **팀원들이 작업한 코드를 통합하고 관리할 때 효율적으로 병합되도록 브랜치 전략을 적용해서 협업 안정성을 높였음**

## 💬 느낀 점

이번 프로젝트는 단순히 기능을 구현하는 것에 그치지 않고, **협업**, **설계**, 그리고 **구조화된 개발의 중요성**을 깊이 체감할 수 있었던 소중한 경험이었습니다.

팀원들과의 협업 과정에서는 Git 브랜치 전략을 기반으로 작업을 진행하며, 이슈 트래킹과 커밋 메시지 컨벤션을 통해 **명확한 커뮤니케이션**과 **히스토리 관리의 중요성**을 직접 체험할 수 있었습니다. 또한 역할 분담을 통해 각자의 책임을 명확히 하며 개발을 진행했고, 이 과정에서 입고 승인, 재고 변경, 재고 이력 저장 등을 각각의 클래스로 분리하면서 **단일 책임 원칙(SRP)**과 모듈화된 설계가 코드의 가독성과 유지보수성에 얼마나 큰 영향을 주는지 체감할 수 있었습니다.
각 로직이 독립되어 있어 기능 변경 시 영향 범위를 좁힐 수 있었고, 테스트도 훨씬 수월했습니다.

특히 이번 프로젝트에서는 **ERD 설계의 중요성**을 크게 체감할 수 있었습니다. 단순히 필드 몇 개를 나열하고 끝내는 것이 아니라, 입고/출고, 재고, 재고 이력과 같은 실제 물류 흐름을 고려하여 **현실 세계의 프로세스를 정확하게 반영할 수 있는 데이터 구조를 설계**해야 했습니다. 각 테이블이 어떤 역할을 하는지, 어떤 관계(FK)를 맺어야 하는지, 그리고 `ON DELETE CASCADE`, `SET NULL`과 같은 제약 조건은 어떤 상황에 필요한지를 고민하며, 현실과 DB 설계 간의 연결 고리를 명확히 파악할 수 있었습니다. 이러한 설계 경험을 통해, **무결성**, **확장성**, **유연성**, **성능**까지 고려한 데이터베이스 구조가 얼마나 중요한지를 깊이 이해하게 되었습니다.

또한 **실시간성과 자동화 로직 구현**에도 도전해보았습니다. 입고 승인 시 자동으로 재고를 갱신하고 트리거를 통해 재고 변경 이력을 관리하는 로직을 직접 설계하고 구현하면서, **실제 시스템에서는 사람이 아닌 코드가 얼마나 많은 판단을 대신해줘야 하는지**, 그리고 이를 가능하게 하려면 **명확한 비즈니스 로직 설계와 데이터 흐름에 대한 깊은 이해**가 필요하다는 점을 깨달았습니다.

## 🚀 2차 개발 시 개선 및 기능 추가 계획

1차 개발을 통해 핵심 WMS 흐름(입고, 출고, 유통기한 관리 등)을 설계·구현했으며,

2차 개발에서는 아래와 같은 기능 확장 및 구조 리팩토링을 진행할 예정입니다.

###  기능 및 기술적 개선 사항

- **콘솔 환경 → Web 환경 전환**
    
    현재 CLI 환경으로 동작 중인 프로그램을
    
    **CSR(Client Side Rendering)** 또는 **SSR(Server Side Rendering)** 기반 웹 UI로 리팩토링하여
    
    **접근성과 사용자 경험을 개선**할 예정입니다.
    
- **거리 계산 API를 통한 마이크로 풀필먼트 기능 구현**
    
    창고 ↔ 배송지 간의 거리를 계산하는 API(Kakao, TMap 등)를 활용해
    
    **가장 가까운 허브에서 자동 출고되는 물류 시스템**을 구축할 계획입니다.
    
- **가독성 높은 코드로 리팩토링**
    
    `람다`, `함수형 인터페이스`, `스트림 API` 등을 적용하여
    
    **예측 가능하고 유지보수하기 쉬운 함수형 코드 구조**로 개선할 예정입니다.
    
- **재고 상태 기반 알림 기능 추가**
    - **일정 수량 이하 재고에 대한 자동 알림 기능 도입**
    - **유통기한이 지난 상품은 자동으로 폐기 처리되도록 구현**
        
        → 시스템 실시간성 확보 및 관리자 대응 속도 향상 목적
